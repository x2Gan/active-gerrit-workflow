# install.sh 实现方案

> 目标：设计一个承接本项目源码分发、环境检查、运行配置、Skill 部署和后续升级的 `install.sh`。
>
> 适用仓库：`active-gerrit-workflow`
>
> 关联产物：
>
> - `active-gerrit/`
> - `active-gerrit-workflow/`
> - `.env.example`
> - `active-gerrit/scripts/gerrit_cli.py doctor`
> - `active-gerrit-workflow/scripts/workflow_cli.py doctor`

## 1. 需求理解

`install.sh` 不只是一个文件复制脚本，而是本项目从“源码仓库”到“用户本地可用 Gerrit Skill”的入口。它应承担五类责任：

| 责任 | 说明 |
|---|---|
| 源码分发 | 用户可以通过一条命令下载或更新本仓库源码，类似 Oh My Zsh、nvm、Homebrew 的源码安装模式。 |
| 环境检查 | 安装前检查 Shell、Git、Python、curl/wget、sed、可选工具和 Gerrit 连通性。 |
| 运行配置 | 交互式采集 Gerrit 地址、用户名、HTTP Password，并写入安全的本地配置文件。 |
| Skill 部署 | 用户输入 Skill 目标目录后，自动把 `active-gerrit`、`active-gerrit-workflow` 以软链接或复制方式部署进去。 |
| 一键升级 | `install.sh update` 检查远端源码、依赖、配置、Skill 链接/副本，并完成可控升级。 |

最终体验应尽量接近：

```bash
bash -c "$(curl -fsSL https://raw.githubusercontent.com/<owner>/active-gerrit-workflow/main/install.sh)"
```

以及已安装后的：

```bash
~/.local/share/active-gerrit-workflow/install.sh doctor
~/.local/share/active-gerrit-workflow/install.sh update
```

## 2. 行业实践调研摘要

### 2.1 Oh My Zsh

Oh My Zsh 的安装方式以远程 `install.sh` 为入口，通过 Git 克隆源码并初始化用户配置。它支持环境变量和参数控制行为，例如 `--unattended`、`RUNZSH`、`CHSH`、`KEEP_ZSHRC` 等。脚本流程是先安装源码，再处理配置文件和 Shell 行为，最后输出下一步提示。

参考：

- [Oh My Zsh install.sh](https://github.com/ohmyzsh/ohmyzsh/blob/master/tools/install.sh)
- [Oh My Zsh README](https://github.com/ohmyzsh/ohmyzsh/blob/master/README.md)

可借鉴点：

- 一条远程命令完成安装。
- 默认安装目录可通过环境变量覆盖。
- 交互和无人值守安装分离。
- 已有配置文件不应无提示覆盖。

### 2.2 nvm

nvm 的安装器同时支持安装和更新。README 明确说明安装脚本会克隆仓库到 `~/.nvm`，并尝试把 source 片段写入合适的 profile 文件。它允许通过 `NVM_DIR`、`PROFILE`、`NVM_SOURCE`、版本变量定制安装来源和目录，也允许 `PROFILE=/dev/null` 禁止修改 shell 配置。

参考：

- [nvm README: Install & Update Script](https://github.com/nvm-sh/nvm#install--update-script)
- [nvm install.sh](https://github.com/nvm-sh/nvm/blob/master/install.sh)

可借鉴点：

- 安装和更新共用同一脚本入口。
- 支持自定义安装目录、安装源、版本。
- 可以自动探测 shell profile，也允许用户显式禁用 profile 修改。
- 已安装时转为更新逻辑，而不是重复安装。

### 2.3 Homebrew

Homebrew 的安装器强调“解释将要做什么，然后暂停等待确认”。它支持 `NONINTERACTIVE=1` 用于自动化，并允许通过环境变量替换 Git remote。Homebrew 也很重视前置检查、sudo 权限、PATH 提示和安装后的 next steps。

参考：

- [Homebrew 官网安装说明](https://brew.sh/)
- [Homebrew/install README](https://github.com/Homebrew/install)
- [Homebrew install.sh](https://github.com/Homebrew/install/blob/HEAD/install.sh)

可借鉴点：

- 默认交互模式先展示操作摘要。
- CI 或无人值守模式必须由环境变量显式开启。
- 远端仓库和路径可配置。
- 安装完成后给出清晰 next steps。

### 2.4 asdf

asdf 的文档把“下载可执行程序”和“把 shims 加入 PATH”拆开说明，并针对不同 shell 给出不同配置方式。它也支持自定义数据目录。

参考：

- [asdf Getting Started](https://asdf-vm.com/guide/getting-started.html)

可借鉴点：

- 运行时入口和 shell 初始化配置应分开处理。
- 不同 shell 的 profile 路径不同，自动修改要谨慎。
- 自定义数据目录需要明确。

### 2.5 XDG Base Directory

Linux/Unix CLI 工具常用 XDG 目录放置配置、数据、缓存和状态。规范定义了默认 fallback：

- `XDG_CONFIG_HOME` 默认 `$HOME/.config`
- `XDG_DATA_HOME` 默认 `$HOME/.local/share`
- `XDG_CACHE_HOME` 默认 `$HOME/.cache`
- `XDG_STATE_HOME` 默认 `$HOME/.local/state`

参考：

- [XDG Base Directory Specification](https://specifications.freedesktop.org/basedir-spec/0.8/)

可借鉴点：

- 源码、配置、缓存、状态文件分目录放置。
- 凭据配置不放仓库源码目录，避免误提交。
- 配置文件权限应限制为 `0600`。

### 2.6 Shell 脚本质量实践

复杂安装脚本建议纳入 ShellCheck 和 Bats 测试。ShellCheck 用于静态发现 shell 脚本常见问题；Bats 是 Bash 脚本测试框架。

参考：

- [ShellCheck](https://github.com/koalaman/shellcheck)
- [bats-core](https://github.com/bats-core/bats-core)

可借鉴点：

- `install.sh` 应有静态检查。
- 安装、更新、参数解析、配置写入、Skill 部署都应有自动化测试。
- 测试应覆盖交互模式和 `NONINTERACTIVE=1`。

## 3. 设计原则

1. 源码是权威安装产物：默认克隆本仓库，而不是只下载单个脚本。
2. 安装过程幂等：重复执行不会破坏已有源码、配置和 Skill 部署。
3. 配置和源码分离：源码放数据目录，凭据放配置目录，缓存放缓存目录。
4. 交互优先安全：默认展示计划并要求确认；无人值守必须显式传入必要变量。
5. 不默认提权：缺依赖时给出安装建议；自动安装系统包必须 `--install-deps` 或 `--yes`。
6. 不泄露敏感信息：密码、token、cookie、Authorization header 永不输出。
7. Skill 部署可选择：默认软链接，兼容复制模式。
8. 更新可回滚：更新前记录版本和配置，更新失败提示如何回到旧 revision。

## 4. 安装目录与文件布局

推荐遵循 XDG 目录：

| 类型 | 默认路径 | 可覆盖变量 | 内容 |
|---|---|---|---|
| 源码目录 | `${XDG_DATA_HOME:-$HOME/.local/share}/active-gerrit-workflow` | `ACTIVE_GERRIT_WORKFLOW_HOME` | 克隆后的完整仓库源码。 |
| 配置目录 | `${XDG_CONFIG_HOME:-$HOME/.config}/active-gerrit-workflow` | `ACTIVE_GERRIT_WORKFLOW_CONFIG_DIR` | 运行环境变量文件、安装元数据。 |
| 配置文件 | `$CONFIG_DIR/env` | `ACTIVE_GERRIT_WORKFLOW_ENV_FILE` | Gerrit 连接配置，权限 `0600`。 |
| 缓存目录 | `${XDG_CACHE_HOME:-$HOME/.cache}/active-gerrit-workflow` | `ACTIVE_GERRIT_WORKFLOW_CACHE_DIR` | 安装器缓存、下载缓存。 |
| 状态目录 | `${XDG_STATE_HOME:-$HOME/.local/state}/active-gerrit-workflow` | `ACTIVE_GERRIT_WORKFLOW_STATE_DIR` | 安装日志、最后安装版本。 |
| 用户 bin | `$HOME/.local/bin` | `ACTIVE_GERRIT_WORKFLOW_BIN_DIR` | 可选 launcher。 |
| Skill 目标目录 | `${CODEX_HOME:-$HOME/.codex}/skills` | `ACTIVE_GERRIT_SKILL_DIR` | 用户输入或默认的 Skill 安装目录。 |

源码目录中保持当前仓库结构：

```text
active-gerrit-workflow/
├── active-gerrit/
├── active-gerrit-workflow/
├── doc/
├── tests/
├── .env.example
├── requirements.txt
└── install.sh
```

配置文件示例：

```bash
# Generated by active-gerrit-workflow install.sh.
export ACTIVE_GERRIT_WORKFLOW_HOME="$HOME/.local/share/active-gerrit-workflow"
export ACTIVE_GERRIT_HOME="$HOME/.local/share/active-gerrit-workflow/active-gerrit"

export GERRIT_BASE_URL="https://gerrit.example.com"
export GERRIT_AUTH_TYPE="basic"
export GERRIT_USERNAME="alice"
export GERRIT_HTTP_PASSWORD="********"

export GERRIT_VERIFY_SSL="true"
export GERRIT_TIMEOUT_SECONDS="30"
export GERRIT_DEFAULT_NOTIFY="OWNER_REVIEWERS"
export GERRIT_CACHE_DIR="$HOME/.cache/active-gerrit-workflow/gerrit"
```

注意：

- `env` 文件必须 `chmod 600`。
- 不把真实密码写入仓库根目录 `.env`。
- `.env.example` 仍作为开发样例保留。
- 如果用户拒绝保存密码，则允许写入不含 `GERRIT_HTTP_PASSWORD` 的 env，并在 doctor 中提示用户用 shell 环境注入。

## 5. 命令设计

`install.sh` 推荐使用子命令：

```text
install.sh [install] [options]
install.sh doctor [options]
install.sh deploy-skill [options]
install.sh config [options]
install.sh update [options]
install.sh status [options]
install.sh uninstall [options]
install.sh help
```

### 5.1 全局选项

| 选项 | 环境变量 | 说明 |
|---|---|---|
| `--repo-url <url>` | `ACTIVE_GERRIT_WORKFLOW_REPO` | 源码仓库地址。 |
| `--ref <ref>` | `ACTIVE_GERRIT_WORKFLOW_REF` | 安装分支、tag 或 commit，默认 `main`。 |
| `--install-dir <path>` | `ACTIVE_GERRIT_WORKFLOW_HOME` | 源码安装目录。 |
| `--config-file <path>` | `ACTIVE_GERRIT_WORKFLOW_ENV_FILE` | 运行配置文件路径。 |
| `--skill-dir <path>` | `ACTIVE_GERRIT_SKILL_DIR` | Skill 目标目录。 |
| `--skill-mode symlink|copy` | `ACTIVE_GERRIT_SKILL_MODE` | Skill 部署方式，默认 `symlink`。 |
| `--non-interactive` | `NONINTERACTIVE=1` | 禁止交互，缺参数直接失败。 |
| `--yes` | `YES=1` | 对安全的确认步骤自动回答 yes。 |
| `--install-deps` | `ACTIVE_GERRIT_INSTALL_DEPS=1` | 尝试通过系统包管理器安装缺失依赖。 |
| `--no-profile` | `PROFILE=/dev/null` | 不修改 shell profile。 |
| `--profile <path>` | `PROFILE=<path>` | 指定 shell profile。 |
| `--force` | `FORCE=1` | 允许覆盖可备份文件或重建 Skill 目标。 |
| `--verbose` | `VERBOSE=1` | 输出调试信息，但仍需脱敏。 |

### 5.2 `install`

默认子命令。负责完整首次安装：

1. 解析参数和环境变量。
2. 检查基础依赖：`bash`、`git`、`python3`、`curl` 或 `wget`、`sed`。
3. 展示安装计划。
4. 克隆或更新源码目录。
5. 引导写入运行配置。
6. 部署 Skill。
7. 运行 `doctor`。
8. 输出 next steps。

幂等规则：

- 源码目录不存在：执行 `git clone`。
- 源码目录存在且是本仓库 Git repo：转入更新或校验。
- 源码目录存在但不是 Git repo：失败并提示换目录或 `--force` 备份。
- 配置文件存在：默认读取旧值作为交互默认值，不覆盖密码，写入前创建 `.bak.<timestamp>`。
- Skill 目标已是正确软链接：跳过。
- Skill 目标是旧副本：根据 `--skill-mode` 同步或提示。
- Skill 目标是用户自有目录：不覆盖，除非 `--force`，且先备份。

### 5.3 `doctor`

负责检查当前安装可用性，分三层：

| 层级 | 检查项 |
|---|---|
| 安装器层 | bash 版本、安装目录、配置目录、PATH、源码 Git 状态、Skill 目标。 |
| 基础 Skill 层 | 执行 `python active-gerrit/scripts/gerrit_cli.py doctor`。 |
| Workflow 层 | 执行 `python active-gerrit-workflow/scripts/workflow_cli.py doctor`。 |

`doctor` 输出应适合人读，也可选 JSON：

```bash
install.sh doctor
install.sh doctor --json
```

推荐检查清单：

| 检查 | 必需 | 失败处理 |
|---|---|---|
| `bash` | 是 | 提示使用 `bash install.sh`。 |
| `git` | 是 | 提示安装；`--install-deps` 时尝试安装。 |
| `python3 >= 3.9` | 是 | 提示安装 Python 3.9+。 |
| `curl` 或 `wget` | 是 | 远程安装和连通性检查需要。 |
| `sed` | 是 | Gerrit XSSI 示例和兼容处理需要。 |
| `jq` | 否 | 标记 warning。 |
| `openssl` | 否 | 标记 warning。 |
| `ssh` | 否 | Git/Gerrit SSH 工作流 warning。 |
| `rg` | 否 | 开发体验 warning。 |
| `shellcheck` | 否 | 开发/CI warning。 |
| `bats` | 否 | 安装器测试 warning。 |
| 配置文件权限 `0600` | 是 | 自动修复或提示。 |
| Gerrit URL 格式 | 是 | 提示输入 `http(s)://...`。 |
| Gerrit `whoami` | 是 | 透传基础 CLI 的脱敏错误。 |
| Skill `SKILL.md` 存在 | 是 | 重新部署 Skill。 |

### 5.4 `config`

负责运行配置写入或更新。

交互式字段：

| 字段 | 环境变量 | 交互行为 |
|---|---|---|
| Gerrit Web 地址 | `GERRIT_BASE_URL` | 必填；默认使用已有配置值。 |
| 鉴权类型 | `GERRIT_AUTH_TYPE` | 默认 `basic`；第一阶段只建议 basic。 |
| 用户名 | `GERRIT_USERNAME` | Basic Auth 必填。 |
| HTTP Password | `GERRIT_HTTP_PASSWORD` | 使用 `read -s` 静默输入；可选择不保存。 |
| TLS 验证 | `GERRIT_VERIFY_SSL` | 默认 `true`。 |
| 超时时间 | `GERRIT_TIMEOUT_SECONDS` | 默认 `30`。 |
| 缓存目录 | `GERRIT_CACHE_DIR` | 默认 `$CACHE_DIR/gerrit`。 |

非交互模式：

```bash
NONINTERACTIVE=1 \
GERRIT_BASE_URL="https://gerrit.example.com" \
GERRIT_USERNAME="alice" \
GERRIT_HTTP_PASSWORD="..." \
bash install.sh install
```

规则：

- `NONINTERACTIVE=1` 下缺必填配置直接失败。
- 密码不允许通过命令行参数传入，只允许环境变量或交互输入。
- 输出摘要时显示 `GERRIT_HTTP_PASSWORD=<redacted>`。
- 配置写入使用临时文件 + 原子 `mv`。

### 5.5 `deploy-skill`

负责把本仓库中的 Skill 部署到用户指定目录。

源目录：

```text
$INSTALL_DIR/active-gerrit
$INSTALL_DIR/active-gerrit-workflow
```

目标目录：

```text
$ACTIVE_GERRIT_SKILL_DIR/active-gerrit
$ACTIVE_GERRIT_SKILL_DIR/active-gerrit-workflow
```

默认目标：

```bash
${CODEX_HOME:-$HOME/.codex}/skills
```

部署模式：

| 模式 | 行为 | 适用场景 |
|---|---|---|
| `symlink` | `ln -sfn <source> <target>` | 默认；源码更新后 Skill 自动跟随。 |
| `copy` | 复制完整 Skill 目录，排除缓存和 pyc | 目标系统不支持软链接，或需要独立副本。 |

复制模式建议优先使用 `rsync -a --delete`；如果没有 `rsync`，回退到 `cp -R` + 清理旧目录。

排除项：

```text
__pycache__/
*.pyc
.cache/
.git/
.DS_Store
```

冲突处理：

| 目标状态 | 默认行为 |
|---|---|
| 不存在 | 创建。 |
| 正确软链接 | 保持。 |
| 错误软链接 | 提示；`--force` 时重建。 |
| 由本安装器复制的目录 | copy 模式下同步。 |
| 用户自有目录 | 不覆盖；提示备份路径；`--force` 时备份后替换。 |

建议写入部署元数据：

```text
$CONFIG_DIR/install-state
```

内容示例：

```bash
INSTALL_DIR="/home/alice/.local/share/active-gerrit-workflow"
SKILL_DIR="/home/alice/.codex/skills"
SKILL_MODE="symlink"
INSTALLED_REF="main"
INSTALLED_COMMIT="abc1234"
INSTALLED_AT="2026-05-11T10:00:00+08:00"
```

### 5.6 `update`

负责一键升级源码、检查依赖和刷新 Skill 部署。

流程：

1. 读取 `$CONFIG_DIR/install-state`。
2. 确认源码目录是 Git repo。
3. 检查 working tree 是否干净。
4. 获取当前 `HEAD` 和远端目标 revision。
5. 如果没有更新，仍运行依赖检查和 Skill 校验。
6. 如果有更新，展示版本变化并确认。
7. 执行 `git fetch --tags --prune`。
8. 按安装策略 checkout：
   - branch 安装：`git pull --ff-only`
   - tag/commit 安装：`git checkout --detach <ref>`
9. 如有 `requirements.txt` 且未来引入依赖，执行依赖检查或提示。
10. 重新执行 `deploy-skill`：
    - symlink 模式：确认链接仍正确。
    - copy 模式：重新同步目录。
11. 运行安装器 `doctor`。
12. 写入新 commit 到 install-state。

脏工作区策略：

| 情况 | 行为 |
|---|---|
| working tree 干净 | 正常更新。 |
| 有用户改动 | 默认停止，提示备份或提交。 |
| `--allow-dirty` | 允许继续，但先打印风险并跳过自动 reset。 |
| `--force` | 仍不执行 `reset --hard`；只允许备份后重新 clone 到新目录。 |

更新失败回滚：

- 更新前记录 `previous_head`。
- 如果 `git pull --ff-only` 失败，不自动 reset。
- 输出手动恢复命令：

```bash
cd "$INSTALL_DIR"
git checkout <previous_head>
install.sh deploy-skill
install.sh doctor
```

### 5.7 `status`

输出当前安装摘要：

- install dir
- repo url
- current branch/ref/commit
- config file
- skill dir
- skill mode
- Skill 链接状态
- Gerrit base URL
- 最近一次 doctor 结果

### 5.8 `uninstall`

建议作为 P2 功能。默认只展示将删除的内容并要求确认：

- Skill 链接或由安装器管理的副本。
- 可选删除源码目录。
- 可选删除配置文件。
- 可选删除缓存和状态。
- 不默认修改 shell profile；如果安装器曾写入 source block，可提示用户删除或自动删除受控 block。

## 6. 远程安装入口设计

推荐官方安装命令：

```bash
bash -c "$(curl -fsSL https://raw.githubusercontent.com/<owner>/active-gerrit-workflow/main/install.sh)"
```

兼容 `wget`：

```bash
bash -c "$(wget -qO- https://raw.githubusercontent.com/<owner>/active-gerrit-workflow/main/install.sh)"
```

建议也提供更安全的手动模式：

```bash
curl -fsSLO https://raw.githubusercontent.com/<owner>/active-gerrit-workflow/main/install.sh
less install.sh
bash install.sh
```

安装器中的默认变量：

```bash
DEFAULT_REPO_URL="https://github.com/<owner>/active-gerrit-workflow.git"
DEFAULT_REF="main"
DEFAULT_INSTALL_DIR="${XDG_DATA_HOME:-$HOME/.local/share}/active-gerrit-workflow"
```

远端仓库覆盖时，参考 nvm 的实践打印安全提示：

```text
Warning: ACTIVE_GERRIT_WORKFLOW_REPO is not the default repository.
Verify this is intentional before continuing.
```

## 7. 脚本内部结构建议

`install.sh` 建议使用 Bash，而不是 POSIX sh。原因是需要数组、`read -s`、更稳的参数处理和 trap。

文件头：

```bash
#!/usr/bin/env bash
set -Eeuo pipefail
IFS=$'\n\t'
```

核心函数分组：

```text
main
  parse_args
  load_defaults
  dispatch_command

logging
  info / warn / error / die
  redact
  confirm

platform
```

默认只检查和提示，不自动安装系统包。

包管理器映射：

| 系统/工具 | 命令示例 |
|---|---|
| Debian/Ubuntu | `sudo apt-get update && sudo apt-get install -y git curl python3 sed` |
| Fedora | `sudo dnf install -y git curl python3 sed` |
| RHEL/CentOS | `sudo yum install -y git curl python3 sed` |

## 12. Shell Profile 与 Launcher

有两种运行时接入方式。

### 12.1 推荐：生成 launcher

在 `$BIN_DIR` 下生成：

```text
active-gerrit
active-gerrit-workflow
active-gerrit-install
```

示例 `active-gerrit`：

```bash
#!/usr/bin/env bash
set -Eeuo pipefail
CONFIG_FILE="${ACTIVE_GERRIT_WORKFLOW_ENV_FILE:-$HOME/.config/active-gerrit-workflow/env}"
# shellcheck disable=SC1090
source "$CONFIG_FILE"
exec python3 "$ACTIVE_GERRIT_HOME/scripts/gerrit_cli.py" "$@"
```

优点：

- 不需要每次手动 source env。
- 不强制修改用户 shell profile。
- Agent 或用户都可以直接调用稳定命令。

### 12.2 可选：修改 shell profile

只有用户确认时，向 profile 写入受控 block：

```bash
# >>> active-gerrit-workflow >>>
export ACTIVE_GERRIT_WORKFLOW_ENV_FILE="$HOME/.config/active-gerrit-workflow/env"
[ -r "$ACTIVE_GERRIT_WORKFLOW_ENV_FILE" ] && . "$ACTIVE_GERRIT_WORKFLOW_ENV_FILE"
# <<< active-gerrit-workflow <<<
```

规则：

- 自动探测 `.zshrc`、`.zprofile`、`.bashrc`、`.bash_profile`、`.profile`。
- `PROFILE=/dev/null` 时不修改。
- 如果 block 已存在，更新 block，不重复追加。
- profile 中不直接写密码，只 source 配置文件。

## 13. 安全设计

必须满足：

1. 所有敏感值输出前脱敏。
2. `GERRIT_HTTP_PASSWORD` 不通过命令行参数传递。
3. 配置目录权限尽量 `0700`，配置文件权限 `0600`。
4. 远端源码仓库被覆盖时打印醒目 warning。
5. 安装计划在交互模式下先展示再执行。
6. 自动安装依赖和覆盖文件必须显式确认。
7. 不执行破坏性 Git 命令。
8. update 前检查源码目录 remote 是否符合预期。
9. 对复制模式的 Skill 目录，替换前先备份。
10. 日志中不记录密码、token、cookie、Authorization header。

脱敏函数建议：

```bash
redact() {
  local text="${1:-}"
  for secret in "${GERRIT_HTTP_PASSWORD:-}" "${GERRIT_BEARER_TOKEN:-}" "${GERRIT_ACCESS_TOKEN:-}"; do
    if [ -n "$secret" ]; then
      text="${text//$secret/<redacted>}"
    fi
  done
  printf '%s\n' "$text"
}
```

## 18. 关键决策建议

1. `install.sh` 用 Bash 实现，明确要求 `bash`，不要追求 POSIX sh。
2. 首版默认只软链接 Skill，复制模式放 P1，但接口先预留。
3. 凭据配置写到 XDG config 下的 `env`，不要写仓库 `.env`。
4. 生成 launcher，降低用户和 Agent 调用 Python 脚本的心智负担。
5. update 不处理脏工作区，不自动 reset；把源码目录视为可读安装产物但尊重用户修改。
6. 依赖自动安装必须是显式开关，避免安装器意外修改系统。
7. `doctor` 复用现有 Python doctor，安装器只做编排和摘要。
8. 文档和脚本中的远端 repo URL 在仓库确定 GitHub 地址后统一替换 `<owner>`。
