# Review Policies

This file defines local default policy for workflow reports. Treat these as conservative defaults until a repository or team-specific policy exists.

## Decision Rule

- `blocked`: known submit or review blocker exists.
- `warning`: risk exists but the workflow lacks enough evidence to block.
- `pass`: required checks have evidence and no known blocker.
- `needs_human_decision: true`: policy or evidence is missing, ambiguous, or business-specific.

## Review Checklist

Evaluate these areas when evidence is available:

- Correctness: edge cases, error handling, data consistency, concurrency, idempotency.
- Tests: unit, integration, manual verification, regression coverage, CI signal.
- Security: authentication, authorization, secrets, injection, unsafe logging, data exposure.
- Reliability: timeouts, retries, partial failure, rollback, compatibility.
- Maintainability: scope, naming, duplication, unnecessary abstraction, readability.
- API and schema compatibility: backward compatibility, migrations, versioned behavior.
- Observability: logs, metrics, traceability, useful error messages without secrets.
- Documentation: user-facing or operator docs when behavior changes.

## Risk Classification

Low risk:

- Small localized change.
- Tests or review evidence already cover the changed path.
- No production config, auth, permission, migration, or release branch impact.

Medium risk:

- Cross-module behavior, public API, non-trivial control flow, or meaningful test gap.
- Reviewer attention needed before approval.

High risk:

- Auth, permissions, secrets, payment, data migration, destructive operation, release branch, build pipeline, deploy path, or large unclear diff.
- Require explicit human review and stronger evidence.

Blocking risk:

- Failing submit requirements.
- Merge conflict or not mergeable.
- Required label missing.
- Unresolved blocking comments.
- Change is WIP, abandoned, merged, private without expected visibility, or not on the intended branch.
- Submit would include unexpected submitted-together changes.

## Vote Policy

- Do not cast `Code-Review +2` automatically unless the user explicitly asks and the workflow has enough evidence.
- Do not cast negative votes automatically unless the user explicitly asks.
- `Code-Review +1` may be proposed when findings are minor and evidence is adequate.
- `Code-Review 0` is the default for generated review comments.
- Only set `Verified` when there is concrete CI or test evidence.
- If label ranges or permissions are unknown, do not vote; report `needs_human_decision: true`.

## Comment Policy

- Prefer concise, actionable comments tied to changed lines.
- Use unresolved comments for issues that require owner action.
- Use resolved or plain patchset-level comments for summaries, notes, or non-blocking observations.
- Never include credentials, tokens, cookies, raw authorization headers, or private environment values.
- Use autogenerated tags for workflow comments.

## Reviewer Assignment

Default reviewer suggestions may use:

- Project or branch ownership.
- File path ownership.
- Prior reviewers on the change.
- Explicit user-provided team rules.
- Gerrit suggested reviewers.

If no confident reviewer can be selected, suggest candidates instead of adding them and set `needs_human_decision: true`.

## Submit Readiness Policy

A workflow may return `pass` only when all applicable evidence exists:

- Change status is `NEW`.
- Current patch set is the intended patch set.
- Submit requirements are satisfied.
- Required labels are present.
- Mergeable check passes.
- Submitted-together changes are visible and acceptable.
- No unresolved blocking comments remain.
- Test or CI evidence is available when the project requires it.

Missing business-specific rules should not become a silent pass; report a warning or human decision need.

## Release Branch Defaults

Treat release branches as higher risk. For release branches, warn when:

- The change is large or touches config, dependency, build, migration, auth, or deployment paths.
- Test evidence is missing.
- Owner or reviewer is unclear.
- There are unresolved comments or very recent patch sets.
- Submitted-together includes changes outside the intended release scope.

Do not submit release branch changes from workflow automation unless the user explicitly confirms after seeing the report.
