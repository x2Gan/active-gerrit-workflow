# Business Workflows

Use this file when the user asks for a business outcome rather than one Gerrit REST operation. Workflow commands should call `active-gerrit` and return a `WorkflowReport`; they must not reimplement Gerrit auth or REST client behavior.

## Workflow Defaults

- Default to report-only behavior.
- Record every lower-level command in `used_active_gerrit_commands`.
- Separate facts from policy judgment.
- When evidence is missing, set `needs_human_decision: true`.
- Any workflow that posts comments, adds reviewers, submits, rebases, abandons, or changes permissions must inherit the base skill dry-run and confirmation rules.

## My Review Queue

Goal: produce a prioritized list of changes waiting for the current user's review.

Data:

- `query-preset my_open_reviews --limit <n>`
- Optional `get-change` for high-risk or stale items.

Logic:

- Default sort: oldest `updated` first.
- Mark WIP, private, unresolved comments, missing current-user response, stale updates, large file count, and release branch targets.
- Group by project or branch when the user asks for planning.

Report:

- Queue summary.
- Per-change next action: inspect, comment, wait for owner, ask for clarification, or skip WIP.
- Evidence fields: change id, subject, owner, branch, updated time, labels, unresolved count.

## Review Brief

Goal: summarize one change before a human or Agent review.

Data:

- `get-change`
- `list-files`
- `get-diff` for selected files
- `list-comments`
- `list-messages` when conversation history matters

Logic:

- Summarize scope, changed areas, file risk, test impact, unresolved discussion, and likely review order.
- Prefer changed files with high churn, config/build/security names, generated-code anomalies, or previous unresolved comments.
- Do not publish comments automatically.

Report:

- Intent summary.
- Changed file overview.
- Risk areas.
- Suggested files to inspect.
- Open questions and next review steps.

## Pre-Submit Check

Goal: decide whether a change appears ready to submit under technical and local workflow rules.

Data:

- Base `submit --dry-run` or equivalent submit precheck.
- `get-change`
- `get-mergeable`
- `submitted-together`

Logic:

- Block if Gerrit submit requirements fail, mergeable is false, status is not `NEW`, the current patch set is stale, or submitted-together contains unresolved blockers.
- Warn for unresolved comments, very recent patch sets, release branch targets, large risky diffs, missing test evidence, or non-visible submitted-together changes.
- If a rule needs project-specific knowledge that is absent, set `needs_human_decision: true`.

Report statuses:

- `pass`: no known blocker.
- `warning`: no known blocker, but human attention recommended.
- `blocked`: do not submit.

## Release Branch Check

Goal: inspect release branch readiness and pending review risk.

Data:

- `query-changes --query "project:{project} branch:{branch} status:open"`
- `list-branches`
- `get-change` for blockers or high-risk changes.

Logic:

- Highlight open changes, WIP changes, missing labels, unresolved comments, merge conflicts, and dependency chains.
- Treat release branches as higher risk by default.
- Do not submit or abandon changes from this workflow.

## Owner Report

Goal: summarize open review load by owner.

Data:

- `query-changes` by project, branch, owner, or date window.
- Optional `get-change` for stale or blocked changes.

Logic:

- Group by owner.
- Mark stale updates, missing reviewers, missing owner response, and repeated patch set churn.

## Stale Review Report

Goal: find reviews that need attention.

Data:

- `query-changes` for open non-WIP changes.
- `list-comments`
- `list-messages`

Logic:

- Mark no reviewer response, no owner response after comments, long idle time, and unresolved comments older than a threshold.
- Thresholds are local policy; if absent, report evidence without declaring violation.

## Add Standard Reviewers

Goal: suggest or add reviewers from local policy.

Data:

- `get-change`
- `query-accounts`
- `add-reviewer` only after plan confirmation

Logic:

- Use project, branch, file path, ownership hints, and explicit policy.
- If policy does not identify a reviewer confidently, return suggestions and `needs_human_decision: true`.
- Default to dry-run.

## Post Review Summary

Goal: publish a generated summary as a patchset-level comment.

Data:

- Generated review brief or pre-submit report.
- `review` command with `/PATCHSET_LEVEL` comment or message.

Logic:

- Always show the exact message before posting.
- Use `tag: "autogenerated:active-gerrit-workflow"`.
- Keep notify default `OWNER_REVIEWERS` unless user overrides.

## Config Change Review

Goal: give special review attention to `refs/meta/config` or project-policy changes.

Data:

- `get-change`
- `list-files`
- `get-diff`

Logic:

- Flag access rules, label definitions, submit requirements, branch permissions, plugin config, and inherited project changes.
- Prefer recommendation and human review over automatic approval.
